# Режимы работы zapret2 (zapret-modes)

**Когда использовать:** пользователь говорит «настроить режим», «какой режим», «переключи режим», «чем отличаются режимы», «прямой обход», «защита туннеля», «у меня VPN». Или при первом запуске для объяснения рекомендованного режима.

---

## Секция 1: Определение текущего состояния (обязательно)

Запустить:
```bash
bash .claude/skills/zapret-modes/scripts/set-mode.sh status
```

Разобрать JSON-вывод и проверить ключевые поля:

| Поле | Значение | Что делать |
|------|----------|------------|
| `current_mode: none` | Режим не выбран | Предложить выбрать режим (перейти к Секции 3) |
| `current_mode: direct` + `vpn_active: true` | Несоответствие | Предупредить: «Выбран прямой обход, но VPN активен. Если VPN работает — рекомендую переключиться на режим защиты туннеля.» |
| `current_mode: tunnel` + `vpn_active: false` | Несоответствие | Предупредить: «Выбрана защита туннеля, но VPN не обнаружен. Для работы этого режима нужен запущенный VPN-клиент.» |
| `current_mode: direct` + `vpn_active: false` | Норма | Режим соответствует состоянию. Переходить к задаче. |
| `current_mode: tunnel` + `vpn_active: true` | Норма | Режим соответствует состоянию. Переходить к задаче. |
| `recommended_mode` | `direct` или `tunnel` | Использовать при предложении режима пользователю |
| `vpn_client` | имя клиента | Подставить в объяснение схемы (Секция 2) |
| `vpn_protocol_family` | xray-family / wireguard / и т.д. | Использовать в Секции 5 для выбора стратегии туннеля |

## Секция 2: Объяснение режимов (MODE-02)

Когда пользователь спрашивает «чем отличаются режимы?», «что такое прямой обход», «что такое защита туннеля» или просит объяснить — использовать ТОЧНО эти схемы (подставить реальный `{vpn_client}` из set-mode.sh status):

```
## Прямой обход (direct)

Как это работает:
  Браузер → Сетевая карта → [zapret2 модифицирует пакеты] → DPI провайдера → Заблокированный сайт

Что защищает: Обходит блокировку сайтов напрямую.
Когда нужен: Нет VPN, хочешь просто открывать заблокированные сайты.
Стратегия: autohostlist (автоматическое определение) + blockcheck (тестирование обхода).
```

```
## Защита туннеля (tunnel)

Как это работает:
  Браузер → VPN-клиент ({vpn_client}) → Туннель → Сетевая карта → [zapret2 маскирует VPN-протокол] → DPI провайдера → VPN-сервер

Что защищает: Маскирует VPN-трафик, чтобы провайдер не мог его заблокировать.
Когда нужен: Используешь VPN (Throne, Nekoray, AmneziaVPN и др.), но провайдер блокирует VPN-протоколы.
Стратегия: маскировка протокола VPN-клиента через nfqws2.
```

**Важно:** Если `vpn_client: none` или `vpn_client: unknown` — не подставлять «none», а пропустить `({vpn_client})` или написать «(VPN-клиент)».

**Ключевое отличие (простыми словами):**
- Прямой обход — для тех, кто хочет открыть заблокированные сайты (YouTube, Twitter и т.д.)
- Защита туннеля — для тех, кто использует VPN, но сам VPN блокируется провайдером

## Секция 3: Выбор режима (MODE-01 agent flow)

Когда пользователь говорит «настроить режим» или когда `current_mode: none`:

1. Запустить `set-mode.sh status` — получить `vpn_active`, `vpn_client`, `recommended_mode`

2. Выбрать рекомендацию:
   - Если `vpn_active: true` → рекомендовать tunnel, объяснить:
     ```
     Вижу, что у тебя активен VPN ({vpn_client}).
     Рекомендую режим «Защита туннеля» — zapret2 будет маскировать VPN-трафик,
     чтобы провайдер не мог заблокировать твой VPN.
     Установить этот режим? (да/нет)
     ```
   - Если `vpn_active: false` → рекомендовать direct, объяснить:
     ```
     VPN не обнаружен.
     Рекомендую режим «Прямой обход» — zapret2 будет обходить блокировки сайтов напрямую.
     Установить этот режим? (да/нет)
     ```

3. После подтверждения пользователя — выполнить:
   ```bash
   bash .claude/skills/zapret-modes/scripts/set-mode.sh set direct
   ```
   или
   ```bash
   bash .claude/skills/zapret-modes/scripts/set-mode.sh set tunnel
   ```

4. После установки режима:
   - **direct** → «Режим установлен. Теперь настроим стратегию обхода для твоего провайдера. Скажи "настроить стратегию" — я запущу blockcheck и подберу параметры.»
   - **tunnel** → «Режим установлен. Теперь настрою стратегию маскировки VPN-протокола.» → перейти к Секции 5

## Секция 4: Стратегия для прямого обхода (MODE-03, direct)

Для прямого обхода используется стандартная настройка стратегии через blockcheck.

Направить пользователя:
```
Для прямого обхода нужно запустить тест стратегий DPI-обхода.
Скажи мне «настроить стратегию обхода» — я запущу blockcheck и подберу параметры для твоего провайдера.
```

Или перейти к `.claude/skills/zapret-config/SKILL.md` самостоятельно.

**Не дублировать** содержимое zapret-config/SKILL.md — только ссылаться на него.

## Секция 5: Стратегия для защиты туннеля (MODE-03, tunnel)

Задача: провайдер блокирует VPN-протоколы (VLESS, VMess, WireGuard, OpenVPN).
Решение: zapret2 модифицирует первые пакеты VPN-соединения, чтобы DPI не распознал протокол.

Получить `vpn_protocol_family` из `set-mode.sh status`. Выбрать стратегию:

| vpn_protocol_family | NFQWS2_OPT стратегия | Пояснение |
|---------------------|----------------------|-----------|
| `xray-family` (VLESS/VMess/Trojan/SS) | `--filter-tcp=443 --filter-l7=tls <HOSTLIST_NOAUTO> --payload=tls_client_hello --lua-desync=fake:blob=fake_default_tls` | VPN маскируется под TLS — DPI определяет по fingerprint — ложное TLS-рукопожатие запутывает DPI |
| `wireguard` | `--filter-udp=51820 <HOSTLIST_NOAUTO> --lua-desync=fake` | WireGuard использует UDP:51820 — отправить ложный первый пакет |
| `openvpn` | `--filter-tcp=1194 <HOSTLIST_NOAUTO> --lua-desync=fake` и `--filter-udp=1194 <HOSTLIST_NOAUTO> --lua-desync=fake` | OpenVPN использует TCP/UDP:1194 — маскировать оба |
| `amnezia-wg` | `--filter-udp=51820 <HOSTLIST_NOAUTO> --lua-desync=fake` | AmneziaWG — модифицированный WireGuard, тот же порт |
| `hysteria` | `--filter-udp=443 --filter-l7=quic <HOSTLIST_NOAUTO> --lua-desync=fake` | Hysteria использует QUIC |
| `clash` | `--filter-tcp=443 --filter-l7=tls <HOSTLIST_NOAUTO> --payload=tls_client_hello --lua-desync=fake:blob=fake_default_tls` | Clash чаще всего использует TLS-based протоколы |
| `trojan` | `--filter-tcp=443 --filter-l7=tls <HOSTLIST_NOAUTO> --payload=tls_client_hello --lua-desync=fake:blob=fake_default_tls` | Trojan маскируется под HTTPS |
| `naiveproxy` | `--filter-tcp=443 --filter-l7=tls <HOSTLIST_NOAUTO> --payload=tls_client_hello --lua-desync=fake:blob=fake_default_tls` | NaiveProxy использует TLS |
| `unknown` | (fallback — см. ниже) | Протокол не определён |

**При `vpn_protocol_family: unknown`:**
```
Не удалось автоматически определить протокол VPN. Для продолжения нужна информация:
— Какой VPN-клиент ты используешь? (Throne, Nekoray, AmneziaVPN, OpenVPN, WireGuard и др.)
— Если знаешь протокол — укажи: VLESS, VMess, WireGuard, OpenVPN и т.д.

Если не знаешь — могу запустить стандартный тест blockcheck как запасной вариант. Попробовать?
```

**Hostlist для tunnel-режима:** Использовать `<HOSTLIST_NOAUTO>` (не `<HOSTLIST>`) — в режиме туннеля нацеливаемся на конкретный VPN-сервер, а не автоматический список.

Если пользователь не знает IP или домен своего VPN-сервера:
```
Для более точной работы нужен адрес твоего VPN-сервера (IP или домен).
Можно посмотреть в настройках VPN-клиента.
Если не хочешь добавлять адрес — можно временно использовать MODE_FILTER=none,
но тогда zapret2 будет обрабатывать весь UDP/TCP трафик (может замедлить интернет).
```

**Применение стратегии:**
1. `bash .claude/skills/zapret-config/scripts/backup-config.sh` → создать бэкап
2. Показать diff (текущий NFQWS2_OPT vs новый)
3. Запросить подтверждение (протокол безопасности из CLAUDE.md)
4. `sudo python3 .claude/skills/zapret-config/scripts/apply-strategy.py "содержимое_nfqws2_opt"` → записать
5. Перезапустить сервис → проверить статус

**Применить стратегию по процедуре из zapret-config/SKILL.md, Секции 5 и 6.** Не дублировать шаги здесь.

**Важное примечание:** Стратегии для tunnel-режима — это отправные точки. Если стратегия не работает (VPN по-прежнему блокируется), предложить blockcheck как запасной вариант:
```
Стандартная стратегия не помогла. Можно попробовать тест blockcheck для поиска рабочего варианта.
Но учти: blockcheck тестирует обход блокировок сайтов, а не маскировку VPN-протоколов.
Результаты могут не совпасть с нужными параметрами. Попробуем?
```

## Секция 6: Переключение режима (MODE-04)

Когда пользователь говорит «переключи на прямой режим», «переключи на защиту туннеля», «смени режим»:

1. Запустить `set-mode.sh status` — определить `current_mode` и `vpn_active`

2. **Если переключение НА прямой обход (direct) ИЗ tunnel:**
   - Предупредить:
     ```
     Режим «Прямой обход» работает без VPN. Если VPN сейчас активен — его нужно будет
     отключить перед настройкой стратегии, иначе blockcheck будет тестировать через VPN
     и результаты будут некорректными.
     Переключить на прямой обход? (да/нет)
     ```
   - После подтверждения: `set-mode.sh set direct`
   - После установки: «Режим переключён. Теперь нужно настроить стратегию обхода под твоего провайдера. Скажи "настроить стратегию".»

3. **Если переключение НА защиту туннеля (tunnel) ИЗ direct:**
   - Проверить `vpn_active`:
     - `vpn_active: false` → СТОП:
       ```
       VPN не обнаружен. Для режима «Защита туннеля» нужен работающий VPN-клиент.
       Сначала запусти VPN, потом скажи мне «переключить на защиту туннеля» снова.
       ```
     - `vpn_active: true` → продолжить:
       ```
       Обнаружен VPN ({vpn_client}). Переключить на режим «Защита туннеля»? (да/нет)
       ```
       После подтверждения: `set-mode.sh set tunnel` → перейти к Секции 5

4. **Если уже в запрошенном режиме:**
   ```
   Ты уже в режиме {current_mode}. Ничего не изменилось.
   ```

## Секция 7: Anti-patterns

**НЕ запускать blockcheck в режиме tunnel для маскировки VPN**
— blockcheck тестирует обход блокировок сайтов, а не маскировку VPN-протокола. Для tunnel используй готовые стратегии из Секции 5. Blockcheck применяется только как запасной вариант если стандартные стратегии не помогли.

**НЕ переключать режим автоматически без решения пользователя**
— Только явный запрос пользователя («переключи режим», «установить режим X») является основанием для изменения. Даже если VPN включён/отключён — не менять режим самостоятельно.

**НЕ рекомендовать MODE_FILTER=none в режиме direct**
— В прямом обходе всегда использовать autohostlist (правило из CLAUDE.md). MODE_FILTER=none применять только как временное решение в tunnel-режиме когда адрес VPN-сервера неизвестен, с явным объяснением компромисса.

**НЕ предполагать адрес VPN-сервера**
— Если нужен адрес для hostlist в tunnel-режиме — спросить у пользователя. Не угадывать и не использовать случайные IP.

**НЕ использовать `<HOSTLIST>` в tunnel-режиме**
— В tunnel-режиме всегда `<HOSTLIST_NOAUTO>`. Авто-список предназначен для прямого обхода сайтов, а не для маскировки VPN-трафика.

**НЕ запускать set-mode.sh без проверки вывода**
— Всегда разбирать вывод: проверять поля `current_mode`, `vpn_active`, `vpn_protocol_family` перед принятием решений.
